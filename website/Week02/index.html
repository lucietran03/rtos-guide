Processes vs Threads

Foundation Concepts

Trước khi đi sâu vào thread programming, cần hiểu sự khác nhau giữa processes và threads - đây là quyết định kiến trúc quan trọng trong real-time systems.

Process

Ý nghĩa: Một chương trình đang chạy với vùng nhớ riêng biệt, hoàn toàn tách biệt với các chương trình khác.

Hình dung: Như các căn hộ trong chung cư - mỗi căn có địa chỉ riêng, có tường ngăn cách, không ai vào được căn của người khác.

Vai trò trong hệ thống: Đảm bảo an toàn - nếu một chương trình bị lỗi và crash, nó không làm ảnh hưởng đến các chương trình khác.

Ví dụ thực tế: Khi bạn mở Word và Chrome cùng lúc, đó là 2 processes riêng biệt. Nếu Chrome crash, Word vẫn chạy bình thường.

Thread

Ý nghĩa: Một luồng công việc trong cùng một chương trình, có thể chia sẻ dữ liệu với các luồng khác.

Hình dung: Như những người sống chung trong một căn hộ - cùng dùng chung tủ lạnh, phòng khách, nhưng mỗi người có giường riêng để ngủ.

Vai trò trong hệ thống: Cho phép một chương trình làm nhiều việc cùng lúc - ví dụ vừa tải file, vừa hiển thị giao diện, vừa phát nhạc.

Ví dụ thực tế: Khi bạn xem video YouTube, có một thread hiển thị hình ảnh, một thread phát âm thanh, một thread tải dữ liệu từ internet.

Aspect Process Thread Khi nào sử dụng
Memory Sharing Isolated (safe) Shared (fast but risky) Process cho an toàn, Thread cho tốc độ
Communication IPC required Direct memory access Process cho hệ thống phân tán, Thread cho cục bộ
Creation Cost Expensive (~ms) Cheap (~μs) Process cho tác vụ dài hạn, Thread cho tác vụ thường xuyên
Fault Tolerance High isolation One crash kills all Process cho hệ thống quan trọng, Thread cho hiệu suất

Giải thích chi tiết bảng so sánh

Tại sao cần hiểu bảng này?

Đây là decision matrix giúp bạn quyết định khi nào dùng Process vs Thread trong real applications. Mỗi dòng giải thích một khía cạnh quan trọng.

1. Memory Sharing (Chia sẻ bộ nhớ)

Process - Isolated (an toàn):

Ý nghĩa: Mỗi process có vùng nhớ riêng biệt, hoàn toàn tách biệt.

Hình dung: Như mỗi gia đình sống ở nhà riêng - không ai có thể vào lấy đồ nhà người khác.

Kết quả: An toàn nhưng không thể share data trực tiếp.

Thread - Shared (nhanh nhưng rủi ro):

Ý nghĩa: Các threads trong cùng process dùng chung vùng nhớ.

Hình dung: Như nhiều người sống chung một nhà - dùng chung tủ lạnh, có thể lấy đồ của nhau.

Kết quả: Nhanh vì share data dễ dàng, nhưng có thể xung đột.

Khi nào chọn gì:

Process: Khi cần an toàn tuyệt đối (banking software, medical devices)

Thread: Khi cần chia sẻ data thường xuyên (video player: một thread decode, một thread hiển thị)

2. Communication (Giao tiếp)

Process - IPC required:

Ý nghĩa: Processes muốn nói chuyện phải dùng Inter-Process Communication (pipes, message queues, sockets).

Hình dung: Như hai gia đình muốn nói chuyện phải gọi điện thoại hoặc gửi thư.

Kết quả: Phức tạp nhưng có thể giao tiếp qua mạng.

Thread - Direct memory access:

Ý nghĩa: Threads giao tiếp bằng cách đọc/ghi trực tiếp vào shared memory.

Hình dung: Như những người trong cùng nhà để lời nhắn trên tủ lạnh.

Kết quả: Rất nhanh nhưng chỉ trong cùng một chương trình.

Khi nào chọn gì:

Process: Hệ thống phân tán (web server + database server)

Thread: Trong cùng một ứng dụng (text editor: một thread typing, một thread spell-check)

3. Creation Cost (Chi phí tạo)

Process - Expensive (~ms):

Ý nghĩa: Tạo process mới mất nhiều thời gian (milliseconds) vì OS phải setup memory space, file handles, etc.

Hình dung: Như xây một ngôi nhà mới - cần thời gian setup nền móng, điện nước, đồ đạc.

Thread - Cheap (~μs):

Ý nghĩa: Tạo thread mới rất nhanh (microseconds) vì chỉ cần thêm một execution context.

Hình dung: Như thêm một người vào nhà hiện tại - chỉ cần thêm giường, còn lại dùng chung.

Khi nào chọn gì:

Process: Tác vụ dài hạn, ít tạo mới (database server chạy cả ngày)

Thread: Tác vụ thường xuyên, tạo nhiều (web server tạo thread cho mỗi request)

4. Fault Tolerance (Khả năng chịu lỗi)

Process - High isolation:

Ý nghĩa: Nếu một process crash, các process khác vẫn chạy bình thường.

Hình dung: Như một nhà cháy, các nhà khác vẫn an toàn.

Thread - One crash kills all:

Ý nghĩa: Nếu một thread crash (segmentation fault, null pointer), toàn bộ process chết.

Hình dung: Như một người trong nhà gây cháy, cả nhà đều bị ảnh hưởng.

Khi nào chọn gì:

Process: Hệ thống quan trọng (flight control, nuclear plant) - một lỗi không được làm sập cả hệ thống

Thread: Khi cần hiệu suất cao và có thể chấp nhận risk (game, media player)

Ví dụ thực tế Decision Making

Ứng dụng Chọn gì Lý do chính Trade-off chấp nhận
Web Browser Process cho mỗi tab Fault tolerance - một tab crash không làm mất tabs khác Tốn memory nhiều hơn
Video Game Threads cho graphics, AI, physics Cần share data liên tục + hiệu suất cao Một thread lỗi có thể crash game
Database Server Process cho mỗi client connection Isolation - một client lỗi không ảnh hưởng clients khác IPC phức tạp cho shared data
Text Editor Threads cho UI, auto-save, spell-check Cần share document data + responsive UI Phải cẩn thận với thread synchronization
Golden Rule cho Decision

Câu hỏi quyết định:

  1. Safety first: Lỗi có thể gây thiệt hại nghiêm trọng không? → Process
  2. Performance critical: Cần tốc độ cực cao và share data nhiều? → Thread
  3. Distributed system: Chạy trên nhiều máy khác nhau? → Process
  4. Frequent creation: Tạo/hủy thường xuyên? → Thread

Thread Lifecycle

Quan trọng cho Programming

Hiểu thread states và transitions là rất quan trọng để debug các vấn đề concurrency trong labs.

Thread States trong QNX

READY State (Sẵn sàng)

Ý nghĩa: Thread đã chuẩn bị xong mọi thứ và sẵn sàng chạy, nhưng đang chờ đến lượt sử dụng CPU.

Hình dung: Như học sinh đã chuẩn bị bài xong, giơ tay chờ cô gọi lên bảng - sẵn sàng nhưng phải chờ lượt.

Vai trò trong hệ thống: Đảm bảo công bằng - các thread chờ theo thứ tự ưu tiên, thread quan trọng hơn sẽ được chạy trước.

RUNNING State (Đang chạy)

Ý nghĩa: Thread đang thực sự làm việc, đang sử dụng CPU để chạy code.

Hình dung: Như học sinh đang đứng ở bảng giải bài - đang có quyền sử dụng bảng và sự chú ý của cả lớp.

Vai trò trong hệ thống: Thực hiện công việc thực sự của chương trình. Chỉ có một thread chạy trên mỗi CPU core tại một thời điểm.

BLOCKED State (Bị chặn)

Ý nghĩa: Thread đang chờ điều gì đó xảy ra (đọc file, nhận tin nhắn, chờ khóa) nên không thể tiếp tục.

Hình dung: Như học sinh đang chờ cô fotocopy đề bài - không thể làm gì khác được, phải chờ đề về.

Vai trò trong hệ thống: Tiết kiệm tài nguyên CPU - thay vì chờ một cách vô ích, thread nhường CPU cho thread khác.

DEAD State (Đã kết thúc)

Ý nghĩa: Thread đã hoàn thành nhiệm vụ và kết thúc, không còn chạy nữa.

Hình dung: Như học sinh đã nộp bài và ra khỏi phòng thi - không còn tham gia vào hệ thống nữa.

Vai trò trong hệ thống: Cần dọn dẹp để giải phóng bộ nhớ và tài nguyên mà thread đã sử dụng.

pthread Programming Pattern - Tại sao cần và dùng như thế nào

Đây là standard pattern cho Real-Time Systems

Trong real-time programming, ta luôn phải explicitly configure thread behavior để đảm bảo predictable performance.

Giải thích từng bước - TẠI SAO cần

Step 1: Khai báo variables để handle thread
pthread_t thread_id;      // Để reference đến thread sau khi tạo
pthread_attr_t attr;      // Để config cách thread sẽ hoạt động

Tại sao cần: Giống như bạn cần số điện thoại để gọi ai đó, ta cần thread_id để control thread sau này.

Step 2: Configure thread behavior TRƯỚC KHI tạo
pthread_attr_init(&attr);                              // Reset attributes về default
pthread_attr_setschedpolicy(&attr, SCHED_FIFO);       // Chọn scheduling: FIFO = real-time
pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED); // Dùng config ta vừa set

Tại sao cần:

  • SCHED_FIFO: Thread sẽ chạy cho đến khi block hoặc yield - KHÔNG bị time-slice interrupt
  • PTHREAD_EXPLICIT_SCHED: Bắt OS phải dùng config ta vừa set, không inherit từ parent thread
  • Kết quả: Thread behavior predictable, phù hợp real-time
Step 3: Tạo thread với config đã setup
pthread_create(&thread_id, &attr, thread_function, &args);

Giải thích parameters:

  • &thread_id: Nơi lưu ID để reference thread sau này
  • &attr: Config ta vừa setup ở step 2
  • thread_function: Function mà thread sẽ chạy
  • &args: Data truyền vào function
Step 4: Wait for thread completion và cleanup
pthread_join(thread_id, NULL);  // Chờ thread kết thúc và cleanup

Tại sao cần:

  • Đồng bộ: Main thread chờ worker thread hoàn thành
  • Cleanup: OS thu hồi resources của thread
  • Avoid zombie threads: Threads kết thúc nhưng chưa cleanup sẽ waste memory

Ví dụ thực tế trong Labs

Lab 2 Example: Temperature Monitoring
// Lab 2: Tạo thread monitor nhiệt độ cao priority
void* temperature_monitor(void* args) {
    while (1) {
        int temp = read_sensor();
        if (temp > 80) {
            emergency_shutdown();  // Critical task!
        }
        sleep(1);
    }
}

int main() {
    pthread_t temp_thread;
    pthread_attr_t attr;

    // Setup high priority cho critical monitoring
    pthread_attr_init(&attr);
    pthread_attr_setschedpolicy(&attr, SCHED_FIFO);
    pthread_attr_setinheritsched(&attr, PTHREAD_EXPLICIT_SCHED);

    // Set priority 50 (cao hơn default priority 10)
    struct sched_param param;
    param.sched_priority = 50;
    pthread_attr_setschedparam(&attr, ¶m);

    // Tạo thread với high priority config
    pthread_create(&temp_thread, &attr, temperature_monitor, NULL);

    // Main thread làm việc khác...
    do_other_work();

    // Wait for monitoring thread (nếu có exit condition)
    pthread_join(temp_thread, NULL);
}

Tại sao pattern này quan trọng:

  • Temperature monitoring PHẢI có priority cao để không miss critical events
  • SCHED_FIFO đảm bảo thread không bị interrupt bởi time slicing
  • Nếu dùng default config, thread có thể chạy chậm và miss emergency
Trong tất cả labs, pattern này để...
  1. Lab 2: Tạo threads với controlled priority cho tasks khác nhau
  2. Lab 3: Ensure threads accessing shared resources có correct priority
  3. Lab 4: Setup producer/consumer threads với appropriate scheduling
  4. Lab 6: Tạo communication threads với predictable timing

Bottom line: Không dùng pattern này = thread behavior không predictable = không phải real-time system!

Scheduling Policies

Core Real-Time Concept

QNX Golden Rule: Highest priority ready task PHẢI chạy ngay lập tức

Scheduling Policies Comparison

Policy Behavior Priority Range Use Case Real-Time Quality
SCHED_FIFO Run until block/yield 1-63 Hard real-time tasks Deterministic
SCHED_RR Round-robin time slicing 1-63 Time-shared real-time Bounded response
SCHED_OTHER System dependent Variable Non-real-time tasks Best effort

Priority Management

Priority Numbers trong QNX

Range: 1-63 (higher number = higher priority)

Default: 10 cho SCHED_FIFO threads

Critical rule: Priority 63 thread sẽ preempt priority 1 thread immediately

Message Passing

QNX cung cấp hai mechanisms chính cho inter-process communication:

POSIX Message Queues

Ý nghĩa: Hệ thống hộp thư điện tử cho phép các chương trình gửi tin nhắn cho nhau mà không cần chờ đối phương online.

Hình dung: Như hộp thư email - bạn gửi email cho bạn bè, không cần biết họ có trực tuyến không, họ sẽ đọc khi tiện.

Vai trò trong hệ thống: Cho phép các phần của chương trình liên lạc mà không bị đơ - người gửi có thể tiếp tục làm việc khác.

Ví dụ: Phần tải nhạc gửi tin "tải xong" vào hộp thư, phần phát nhạc đọc khi sẵn sàng.

QNX Native Messaging

Ý nghĩa: Hệ thống liên lạc trực tiếp giữa các phần của chương trình, phải chờ phản hồi ngay lập tức.

Hình dung: Như gọi điện thoại - bạn gọi, chờ người kia bắt máy và trả lời, mới cúp máy được.

Vai trò trong hệ thống: Đảm bảo tính thời gian thực - khi cần phản hồi ngay lập tức thì dùng cách này, không được trễ.

Ví dụ: Chương trình điều khiển phanh xe hỏi cảm biến "xe đang đi bao nhanh?" và phải có câu trả lời ngay.

Synchronization Overview

Preview cho Labs 3-6

Khi nhiều threads cùng làm việc, chúng cần "phối hợp" với nhau để tránh xung đột. Đây là preview các công cụ synchronization sẽ học trong labs sau.

Tại sao cần đồng bộ hóa?

Vấn đề Race Condition

Ý nghĩa: Khi nhiều threads cùng truy cập một dữ liệu và kết quả phụ thuộc vào thứ tự thực thi (không đoán trước được).

Hình dung: Như hai người cùng rút tiền từ một tài khoản cùng lúc - nếu không có kiểm soát, có thể rút được nhiều hơn số dư thực tế.

Vai trò trong hệ thống: Gây ra bugs khó tìm vì chỉ xảy ra trong timing conditions đặc biệt.

Ví dụ thực tế: Thread A đọc counter = 100, Thread B cũng đọc counter = 100. Cả hai đều tăng lên 101 và ghi lại. Kết quả: counter = 101 thay vì 102!

Các công cụ đồng bộ hóa

Mutex (Mutual Exclusion)

Ý nghĩa: Một "khóa" chỉ cho phép một thread truy cập vùng dữ liệu tại một thời điểm.

Hình dung: Như khóa phòng vệ sinh - chỉ một người được vào, người khác phải chờ đến khi có người ra và trả chìa khóa.

Vai trò trong hệ thống: Bảo vệ dữ liệu chia sẻ khỏi race conditions bằng cách đảm bảo truy cập độc quyền.

API: pthread_mutex_lock(), pthread_mutex_unlock()

Semaphore

Ý nghĩa: Một "bộ đếm" cho phép một số lượng giới hạn threads truy cập resource cùng lúc.

Hình dung: Như bãi đỗ xe có 5 chỗ - tối đa 5 xe được vào, xe thứ 6 phải chờ đến khi có xe ra.

Vai trò trong hệ thống: Kiểm soát số lượng threads có thể dùng tài nguyên hạn chế đồng thời.

API: sem_wait(), sem_post(), sem_init()

Barrier

Ý nghĩa: Một "checkpoint" buộc tất cả threads phải đợi đến khi mọi thread đều đến điểm này.

Hình dung: Như tour guide chờ tất cả người trong đoàn tập trung tại một điểm trước khi tiếp tục.

Vai trò trong hệ thống: Synchronize multiple threads để đảm bảo chúng di chuyển cùng nhau qua các phases.

Condition Variables

Hình dung: Như chờ đèn xanh tại ngã tư - không liên tục kiểm tra mà ngủ đợi đến khi có tín hiệu.

Vai trò trong hệ thống: Efficient waiting mechanism thay vì busy polling (liên tục kiểm tra).

1. Thread Creation & Scheduling (sched_policy-1.c)

Purpose & Learning Goal

What it demonstrates: How to get và set thread scheduling policies and priorities

Why important: Essential pattern cho tất cả real-time applications

Lab connection: Foundation cho Lab 2 thread programming

Key Code Pattern
struct sched_param param;
int policy;

// Get current scheduling info
sched_getparam(0, ¶m);           // 0 = current thread
policy = sched_getscheduler(0);

// Set new priority
param.sched_priority = 20;           // Higher number = higher priority
sched_setparam(0, ¶m);

// Set new policy
sched_setscheduler(0, SCHED_FIFO, ¶m);

Line-by-line explanation:

  • sched_getparam(0, ¶m): Lấy priority hiện tại của thread này (0 = current thread)
  • sched_getscheduler(0): Lấy scheduling policy hiện tại (FIFO, RR, OTHER)
  • param.sched_priority = 20: Set priority level (1-63, higher = more important)
  • sched_setscheduler(): Change cả policy và priority cùng lúc

2. Message Queue Communication (mq_send/receive-1.c)

POSIX Message Queue Pattern

Sender side (mq_send-1.c):

// 1. Setup queue attributes
struct mq_attr attr;
attr.mq_maxmsg = 100;              // Max 100 messages in queue
attr.mq_msgsize = MESSAGESIZE;     // Each message max size
attr.mq_flags = 0;                 // Blocking mode

// 2. Create/open queue
mqd_t qd = mq_open("/test_queue",
                   O_RDWR | O_CREAT,     // Read/write, create if not exist
                   S_IRUSR | S_IWUSR,    // Permissions for user
                   &attr);

// 3. Send message
char message[] = "Hello from sender";
mq_send(qd, message, strlen(message), 0);  // Priority 0

// 4. Cleanup
mq_close(qd);

Common mistakes to avoid:

  • Không check return values cho errors
  • Forget mq_close() causing resource leaks
  • Wrong message size causing truncation

Receiver side (mq_receive-1.c):

// 1. Open existing queue (read-only)
mqd_t qd = mq_open("/test_queue", O_RDONLY);

// 2. Receive messages in loop
char buffer[MESSAGESIZE];
ssize_t bytes_read;

while ((bytes_read = mq_receive(qd, buffer, MESSAGESIZE, NULL)) > 0) {
    buffer[bytes_read] = '\0';     // Null terminate
    printf("Received: '%s'\n", buffer);

    if (strcmp(buffer, "done") == 0) break;  // Exit condition
}

// 3. Cleanup
mq_close(qd);
mq_unlink("/test_queue");  // Remove queue from system

Key insights:

  • mq_receive() blocks: Wait until message available
  • NULL terminate: Messages không automatically null-terminated
  • mq_unlink(): Only receiver should remove queue

3. QNX Native Messaging (MsgPass-Server/Client-1.c)

Synchronous Client-Server Pattern

Server side patterns:

// 1. Create named channel
name_attach_t *attach = name_attach(NULL, "myserver", 0);
if (!attach) {
    perror("name_attach failed");
    exit(1);
}

// 2. Message receive loop
while (1) {
    rcvid = MsgReceive(attach->chid, &msg, sizeof(msg), NULL);

    if (rcvid == 0) {
        // Pulse received - handle disconnect/unblock
        if (msg.hdr.code == _PULSE_CODE_DISCONNECT) {
            ConnectDetach(msg.hdr.scoid);
        }
    } else if (rcvid > 0) {
        // Message received - process and reply
        process_message(&msg);
        MsgReply(rcvid, EOK, &reply, sizeof(reply));
    } else {
        // Error occurred
        perror("MsgReceive failed");
        break;
    }
}

// 3. Cleanup
name_detach(attach, 0);

Critical server responsibilities:

  • Handle pulses: Client disconnect events phải được xử lý
  • Always reply: Mọi message phải có reply, nếu không client bị hang
  • Error checking: rcvid values indicate different event types

Client side patterns:

// 1. Connect to named server
int server_coid = name_open("myserver", 0);
if (server_coid == -1) {
    perror("name_open failed");
    exit(1);
}

// 2. Send request and wait for reply
request_msg_t request;
reply_msg_t reply;

strcpy(request.data, "Hello server");
request.client_id = getpid();

if (MsgSend(server_coid, &request, sizeof(request),
            &reply, sizeof(reply)) == -1) {
    perror("MsgSend failed");
    exit(1);
}

printf("Server replied: %s\n", reply.data);

// 3. Cleanup
name_close(server_coid);

Client behavior:

  • Blocking call: MsgSend() blocks until server replies
  • Atomic operation: Send request + receive reply trong one call
  • Network transparent: Same API cho local và remote servers
